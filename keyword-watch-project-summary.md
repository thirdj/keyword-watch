# 키워드 감시 알림 서비스 — 프로젝트 정리

관심 키워드를 등록하면 주기적으로 검색해서, 새 결과가 있을 때만 텔레그램/이메일로 알려주는 개인용 도구.

**스택**: Next.js + Neon PostgreSQL + Vercel Cron + Tavily Search API

---

## 핵심 요구사항

- 키워드 5~10개 등록
- 키워드별로 확인 주기 직접 설정 가능
- 검색엔진 직접 선택
- 새 결과 있을 때만 알림 (없으면 무알림)
- 알림 채널: 텔레그램 / 이메일

---

## 진행하며 바뀐 주요 결정들

| 항목 | 처음 계획 | 최종 결정 | 이유 |
|---|---|---|---|
| 검색엔진 | Google Custom Search | Tavily | Google CSE 신규 발급 중단, Bing도 단종. Tavily는 무료(월 1,000건)에 한글 검색 품질도 양호 확인함 |
| 정보 소스 | 검색 API + RSS 병행 | 검색 API만 | RSS는 "출처를 미리 알아야" 쓸 수 있어서, 애초에 "어디서 나올지 모르는 키워드를 찾아주는" 서비스 취지와 모순됨 |
| 신규 판별 방식 | 직전 체크와 비교 | 누적 URL 집합과 비교 | 검색 순위 변동으로 인한 오탐 방지 |
| 위키/블로그류 | (미고려) | 도메인별 content 해시 비교 추가 | 같은 URL이어도 내용이 계속 바뀌는 페이지(나무위키 등) 대응 |
| 최소 확인 주기 | (미정) | 서버에서 1시간 하한 강제 | 무료 크레딧(월 1,000건) 보호 |

---

## 데이터 모델

- `keywords` — 등록된 키워드, 검색엔진, 주기, 활성 여부
- `check_logs` — 매 체크 실행 기록 (결과 해시, 신규 여부)
- `seen_urls` — 키워드별로 지금까지 본 URL 누적, 신규 판별의 기준
- `notifications` — 채널별 발송 이력 (성공/실패 추적)
- `notification_channels` — 텔레그램 chat_id, 이메일 주소
- `volatile_domains` — 콘텐츠가 자주 바뀌는 도메인 목록 (나무위키, 네이버 블로그 등)

## 단계별 구현 현황

### ✅ 1단계 — 데이터 모델
Neon PostgreSQL 스키마 확정 (위 6개 테이블)

### ✅ 2단계 — 검색 API 연동
- `SearchAdapter` 인터페이스로 검색엔진 추상화
- Tavily 어댑터: relevance score 0.7 이상만 채택, 마크다운 노이즈 정제
- Naver 어댑터: 국내 이슈 보완용 병행 옵션

### ✅ 3단계 — 신규 결과 판별
- 키워드별 누적 URL 집합과 비교하는 방식
- 변동성 도메인(위키/블로그)은 content 해시로 추가 검증
- 첫 체크는 baseline만 저장 (알림 폭탄 방지)

### ✅ 4단계 — 스케줄러
- Vercel Cron 30분마다 실행 → `interval_min` 지난 키워드만 골라서 체크
- 키워드별 실패가 전체를 막지 않도록 개별 try/catch
- `CRON_SECRET`으로 엔드포인트 보호

### ✅ 5단계 — 알림 발송
- 텔레그램 봇 (BotFather로 토큰 발급, 무료)
- 이메일 (Resend, 월 3,000건 무료)
- 채널별 발송 이력을 `notifications`에 기록, 한 채널 실패해도 나머지는 계속 진행

### ✅ 6단계 — 프론트엔드
- 키워드 목록 대시보드 (상태 점, 검색엔진 태그, 주기 표시)
- 키워드 추가 모달 — 저장 전 예상 월간 사용량을 실시간으로 보여줌 (무료 한도 대비 %)
- 서버 검증 에러를 그대로 화면에 노출

---

## 남은 단계

### 7단계 — 배포 및 운영
- Vercel 배포, 환경변수 등록 (`TAVILY_API_KEY`, `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`, `RESEND_API_KEY`, `CRON_SECRET`, `DATABASE_URL`)
- Cron 실행 로그 모니터링 — 실패가 반복되면 알림 오게 해두면 좋음

### 8단계 — 예외 처리 다듬기
- Tavily 크레딧 소진 시 대응 (429 응답 처리, 남은 크레딧 `/usage` 엔드포인트로 사전 확인)
- 검색 결과가 아예 없는 키워드 처리
- `volatile_domains` 목록을 실사용하며 계속 보강

## 크레딧 계산 참고

Tavily 무료 한도: 월 1,000건

| 키워드 수 | 확인 주기 | 월 사용량 | 한도 대비 |
|---|---|---|---|
| 10개 | 2시간마다 | 3,600건 | 초과 |
| 10개 | 6시간마다 | 1,200건 | 초과 |
| 10개 | 8시간마다 | 900건 | 90% |
| 8개 | 8시간마다 | 720건 | 72% |

→ 개인 용도로는 **8시간 주기 내외**가 무료 한도 안에서 현실적인 선으로 확인함.
